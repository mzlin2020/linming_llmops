# linming_llmops 开发指导文档

> 本文档是项目的开发计划划分与移植指导。配套总览见根目录 `README.md` 的「开发阶段划分」一节。
>
> 说明：本项目由一个闭源系统的 AI 能力剥离而来。文中「原前端 / 原后端 / 原网关服务」均指该闭源系统对应部分，仅作功能性参照；其内部源码全程只读、不在本仓库公开。

## 一、背景与目标

把现有闭源系统的 AI 能力剥离为一个**可开源、可 Docker 一键部署**的完整 LLMOps 项目（含前端 + 后端）。

- 忠实借鉴原前端与原后端，**排除用户/权限逻辑**及一切不适合开源的部分。
- 前端技术栈 Next.js → 纯 React；后端基本一致（Flask）。
- 每阶段结束都要走完整测试；敏感信息全部抽离为 `.env` 可配置。
- 最终态：用户 `cp .env.example .env` 填好后 `docker compose up` 即可运行。
- 原项目代码全程只读，不得修改。

## 二、已确定的关键决策

| 项 | 决策 | 对开发的影响 |
|---|---|---|
| 身份方案 | **内置轻量登录，无管理员概念**（自建 `account` 表 + 邮箱/账号密码登录） | 需新建独立 `account` 表与登录流程；排除复杂 RBAC；`quota_service` 中 `is_admin` 分支拍平为**统一、可配置**的配额上限 |
| 模型供应商 | **OpenAI 兼容为默认**（`DEFAULT_LLM_PROVIDER=openai`） | Anthropic / 火山 Ark 作为 env 开关的可选插件；图像生成 / TTS 默认关闭，配置后才启用 |
| 文件/图片存储 | **本地磁盘默认**（`STORAGE_ROOT` + `/files` 静态路由） | 替换对原网关服务的 OSS 委托；保留 S3/MinIO 适配器供 env 切换 |
| v1 功能范围 | **核心优先**：应用编排 + 对话(SSE) + 知识库/RAG + 内置/API 工具 + 助手 agent。**工作流编辑器、图像、TTS 后置到 v1.1** | 减少首版前端重写量与风险；工作流引擎(`core/workflow`)可随核心一并移植但 v1 不对外暴露 |

### 按合理默认锁定
- **数据库**：MySQL 8；向量库 Qdrant；缓存/队列 Redis + Celery。
- **前端**：Vite + React + React Router 的 SPA，nginx 静态托管并反代 `/api` 与 SSE。
- **嵌入模型**：默认 `BAAI/bge-small-zh-v1.5`（CPU 友好、约 200MB 内存）；首次启动挂载卷下载，文档提供 `HF_ENDPOINT` 镜像；另提供「内置权重」镜像变体可选。
- **开源协议**：建议 Apache-2.0（待最终确认，属轻量决策）。

## 三、必须改造/解耦的耦合点（移植前必须心里有数）

1. **认证耦合（最核心）**：原后端用共享密钥 JWT 校验原网关的 `user`/`roles`/`permissions` 表；约 12 张 `ai_*` 表带 `user_id → user.id` 外键；处理器把 `current_user` 直接透传给服务；配额按 `is_admin` 分支。
   - 关键位置：原后端的认证中间件与装饰器（`RequireLogin`/`RequirePermission`、`current_user` 接缝）。
2. **迁移基线无法自举**：原始基线迁移硬编码了 `ForeignKeyConstraint([...],["user.id"])`，且历史迁移链假设原网关的 `user`/`roles`/`permissions` 表已存在 → **空库无法 `alembic upgrade`**，必须重建基线。
3. **存储委托原网关**：图像生成与对话附件经 `POST /api/resource/upload-by-url` 委托给原网关服务，本地无对象存储 SDK 路径，必须替换为自托管存储适配器。
4. **火山引擎是中国特有**：Ark 网关 / Seedream 文生图 / 火山 TTS 需中国账号与区域端点，对国际开源用户不友好 → 默认改 OpenAI 兼容，火山降级为可选。
5. **前端 Next 化**：AI 能力已隔离在原前端的 `app/workshop/*` 与 `api/ai/*`（15 个模块），但 SSE 客户端（`lib/sse.ts`，POST-SSE）、axios 客户端（`lib/request.ts`）耦合用户态 store、`NEXT_PUBLIC_*`、Next 文件路由与令牌刷新逻辑，需重写为框架无关。
6. **测试夹具耦合认证**：原后端 `test/conftest.py` 会插入真实 `User` 行并签发原网关格式 JWT → 认证改造后必须先重写夹具，否则任何处理器测试都跑不过。`fake_llm`/`fake_embeddings`/`fake_tts` 等 monkeypatch 夹具保留，避免真实付费调用。

## 四、开发阶段划分

依赖主线：**基础设施 → 认证接缝 →（数据模型 → 迁移基线）→ 核心引擎 → 服务/存储 → 前端 → 一键部署 → 开源打磨 →（v1.1：工作流/图像/TTS）**。每阶段以「累计测试全绿」为出口。

### Phase 0 — 脚手架 + 基础设施 + 决策锁定
- 目标：空仓库可启动，基础设施先跑起来。
- 交付：`backend/`、`frontend/`、`deploy/`、`docs/` 目录骨架；`docker-compose.yml` 仅起 **MySQL 8 / Redis / Qdrant**（带 healthcheck + 命名卷）；以原后端默认配置全量 key 派生 `.env.example`（分组：基础设施 / LLM 供应商 / 嵌入 / 存储 / 配额 / 认证）；CI 骨架（空 `pytest` + `vitest` 能跑）。
- 出口测试：`docker compose ps` 健康；CI 空测试通过。

### Phase 1 — 后端骨架 + 轻量登录认证（替换原网关耦合）
- 目标：Flask 应用壳 + DI(`injector`) + 扩展 + 配置 + 错误信封 + 路由可启动；`/ping` 200。**认证层换成自建轻量登录**。
- 交付：移植 `app/http`、`config/`、`pkg/`、`internal/extension`、`internal/exception`、`internal/router`；重写认证中间件与装饰器；用自建 `account` 服务 + 邮箱/账号密码登录替换原账号/JWT 服务；**保留 `current_user` 接缝**使下游处理器无需改动；移植 `Dockerfile` + `docker/entrypoint.sh`（保留两段式 CPU-torch 构建）。
- 出口测试：先重写 `conftest.py` 的身份夹具；认证单测（匿名 vs 登录、登录/登出/令牌、`RequireLogin` 等价物）通过；应用可启动。

### Phase 2 — 数据模型 + 重建迁移基线（解耦关键）
- 目标：移植 `internal/model/*`，去掉全部 `FK → user.id`，**重建单一干净 Alembic 基线**，空 MySQL 可自举建表。
- 交付：`user_id` 改为指向自建 `account` 表（或保留为带索引的普通列）；删除 `User/Role/Permission` 镜像模型；删掉历史迁移，对最终模型 `flask db migrate` 生成**一条压平的 `0001_initial`** 并人工审阅；`entrypoint.sh` 的 `MIGRATION_ENABLED=true` 能从空库建全量 schema。
- 出口测试：空库 `upgrade` 成功 + `downgrade` 可逆 + `flask db migrate` 无 autogen 漂移；模型层单测通过。

### Phase 3 — 核心 AI 引擎移植（供应商无关）
- 目标：移植 `internal/core/*`，供应商可配置而非火山硬编码。
- 交付：`core/language_model`（默认 OpenAI 兼容；Anthropic/火山/DeepSeek 为 env-gated 可选）、`core/embeddings`、`core/vector_store`(Qdrant)、`core/retrievers`、`core/agent`、`core/tools/{builtin_tools,api_tools}`、`core/file_extractor`、`core/memory`、`core/mcp`；`core/workflow`（8 节点引擎）**可一并移植但 v1 不暴露**；嵌入默认小 bge。
- 出口测试：移植核心层 `test/internal/core/**`，用 `fake_llm`/`fake_embeddings` 不打真实供应商；核心引擎测试全绿。

### Phase 4 — 服务 + 处理器移植（v1 后端功能完整）+ 自托管存储
- 目标：移植 `internal/service`(~28) + `internal/handler`(~21) + `internal/task`(Celery)，全部接到新身份接缝；替换原网关 OSS。
- 交付（**v1 范围**）：app/chat/conversation、dataset/document/segment/indexing/retrieval/keyword/process_rule、api_tool/builtin_tool、assistant_agent、language_model/llm_admin、stats、upload_file、openapi/api_key 的服务+处理器+Celery 任务；**用本地磁盘存储适配器替换上传委托**（接口化，留 S3/MinIO 切换）；`quota_service` 去掉 `is_admin` 分支，改为统一可配置配额；compose 加 Celery worker。
- 暂不在 v1 暴露：图像生成、TTS 处理器（移到 v1.1）。
- 出口测试：移植 `test/internal/handler/**`（真实 DB + 假 LLM/嵌入）；新增存储适配器测试；目标与原用例集对齐（去掉用户/权限与原 OSS 专属用例，改为替换版）。

### Phase 5 — 前端重写（Vite React SPA，v1 范围）
- 目标：把 workshop 的核心 AI 界面重写为纯 React Vite SPA。
- 交付：Vite + React Router 替换 Next 文件路由；移植 `api/ai/*` 中 v1 所需模块到**重写后的 axios 客户端**（去掉 `NEXT_PUBLIC_` 与 Next 令牌存储）与**重写后的 POST-SSE**（`fetch` + `ReadableStream`，对接后端 SSE 帧协议）；移植状态管理、Radix UI + Tailwind(shadcn)、react-hook-form + zod；nginx 静态托管容器。
- **v1 排除**：工作流编辑器（`@xyflow/react`+dagre，最重）、图像、TTS 界面 → v1.1。
- 出口测试：`vitest` + Testing Library 组件测试；对 SSE reader 的脚本化字节流单测（解析/中断/错误帧）；类型检查 + 构建通过。

### Phase 6 — Docker 一键部署 + 集成硬化（v1 全栈）
- 目标：干净主机上 `cp .env.example .env && docker compose up` 端到端可用。
- 交付：定稿 `docker-compose.yml`（mysql/redis/qdrant/backend(gunicorn)/celery-worker/frontend(nginx)，可选 minio）；healthcheck 排序启动；后端启动即迁移；嵌入模型挂载卷下载，`HF_ENDPOINT` 镜像有文档；nginx 反代 `/api` 与 SSE（验证不缓冲）；一条冒烟脚本：建应用 → 对话(SSE) → 建知识库 → RAG 检索。
- 出口测试：组合栈端到端冒烟全过。

### Phase 7 — 开源打磨（v1 发布）
- 目标：成为可信开源项目。
- 交付：`README`（架构、快速上手、`.env` 参照表）、`LICENSE`（待定 Apache-2.0/MIT）、定稿 `.env.example`、`CONTRIBUTING`、供应商/模型配置指南、安全说明（默认凭证、密钥）；清除残留的中国特有默认值与对原网关服务的引用；全量回归一遍。

### Phase 8 — v1.1：工作流 + 图像 + TTS（后置）
- 目标：补齐被后置的高级能力。
- 交付：暴露 `core/workflow` 引擎的处理器 + 前端 `@xyflow/react`+dagre 工作流编辑器（最大子任务，内部排序：客户端/SSE 基建 → 简单模块 → 数据集/对话 → 工作流编辑器最后）；图像生成与 TTS 的供应商可插拔实现（火山为可选，默认关闭，未配置时优雅降级）。
- 出口测试：工作流 8 节点 e2e（建图 → 调试运行）、图像/TTS 在配置后冒烟通过。

## 五、每阶段「完整测试」定义

复用原后端测试套件（集成偏重：真实 DB/Redis/Qdrant + monkeypatch 假供应商）。「完整测试通过」= 截至该阶段的**累计**套件全绿（除真实付费 API 路径保持 mock 外无 skip）。各阶段重点见上表，关键约束：**认证夹具必须在 Phase 1 先重写**，否则后续处理器测试无法通过。

## 六、风险登记

| # | 风险 | 影响 | 缓解 |
|---|---|---|---|
| R1 | 认证解耦波及面广（约 12 表 FK、`current_user` 贯穿、配额依赖 `is_admin`） | 高 | Phase 0 锁定方案；`current_user` 保持稳定接缝；无管理员概念后配额拍平为统一可配置；Phase 1 首先重写 `conftest.py` 夹具 |
| R2 | 火山供应商中国特有 | 中 | 默认 OpenAI 兼容；火山/图像/TTS 默认关闭、env-gated；未配置优雅降级 |
| R3 | 本地嵌入模型撑大镜像/首启下载（torch CPU、bge 权重） | 中-高 | 默认小 bge；挂载卷首启下载 + `HF_ENDPOINT` 镜像；另出「内置权重」镜像变体；沿用两段式 CPU-torch 构建避免拉 CUDA(~6GB) |
| R4 | 纯 React 下 POST-SSE 重写（原耦合用户态 store/令牌刷新） | 中 | Phase 5 早期实现框架无关的 `fetch`+`ReadableStream` reader；脚本化字节流单测；Phase 6 验证 nginx 不缓冲 |
| R5 | 重建 Alembic 基线不干净（历史链假设原网关表已存在） | 高 | Phase 2 去 FK 后一次性重建；人工审阅 `0001_initial`；以「`migrate` 无漂移 + 空库 upgrade 成功」为闸 |
| R6 | 原网关存储耦合阻塞一键部署 | 高 | Phase 4 引入存储适配器接口；默认本地磁盘；替换全部调用点 + 适配器测试 |
| R7 | 前端重写工作量大（workshop 120+ 文件，工作流编辑器 ~32 文件） | 中-高 | v1 已后置工作流/图像/TTS；只移植不重设计 UI 控制范围 |
| R8 | 测试套件依赖真实基础设施与共享密钥 | 中 | CI 在 Phase 0 备好 MySQL/Redis/Qdrant；Phase 1 重写身份夹具；保留假供应商夹具 |

## 八、验证方式（最终）

干净主机执行 `cp deploy/.env.example deploy/.env`（仅填 LLM 凭证）→ `docker compose up` → 走冒烟链路：**注册/登录 → 建应用 → 对话(SSE) → 建知识库并 RAG 检索 → 绑定内置/API 工具让 agent 调用**。后端 `pytest` 全绿、前端 `vitest` + 构建通过，即视为 v1 达标。
